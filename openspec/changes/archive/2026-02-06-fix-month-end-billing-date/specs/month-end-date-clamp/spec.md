## ADDED Requirements

### 需求：公历月周期日期钳位
系统在计算公历月/年周期的下一个到期日时，必须将日期钳位到目标月份的最后一天（当原始日期大于目标月份天数时），而非让 JavaScript Date 自动溢出到下一个月。

#### 场景：月末日期加一个月（目标月天数不足）
- **当** 当前到期日为 2024-01-31，周期为 1 个月
- **那么** 下一个到期日必须为 2024-02-29（闰年 2 月最后一天），而非 2024-03-02

#### 场景：月末日期加一个月（目标月天数充足）
- **当** 当前到期日为 2024-02-29，周期为 1 个月，anchorDay 为 31
- **那么** 下一个到期日必须为 2024-03-31（3 月有 31 天，恢复到 anchorDay）

#### 场景：非月末日期正常计算
- **当** 当前到期日为 2024-01-15，周期为 1 个月
- **那么** 下一个到期日必须为 2024-02-15（无钳位需要）

#### 场景：跨年计算
- **当** 当前到期日为 2024-11-30，周期为 3 个月
- **那么** 下一个到期日必须为 2025-02-28（2025 年非闰年，2 月最多 28 天）

### 需求：anchorDay 锚定日机制
系统必须在订阅对象中维护 anchorDay 字段，记录用户期望的月内到期日，以便在月末钳位后的后续周期中恢复到正确日期。

#### 场景：新建订阅自动设置 anchorDay
- **当** 用户创建订阅，起始日期为 2024-01-31
- **那么** 系统必须将 anchorDay 设置为 31

#### 场景：现有订阅兼容（无 anchorDay 字段）
- **当** 系统处理一个没有 anchorDay 字段的历史订阅，到期日为 2024-03-15
- **那么** 系统必须使用到期日的 day（15）作为默认 anchorDay

#### 场景：编辑到期日更新 anchorDay
- **当** 用户手动将到期日修改为 2024-05-31
- **那么** 系统必须将 anchorDay 更新为 31

### 需求：年周期闰年日期钳位
系统在计算公历年周期时，必须正确处理 2 月 29 日（闰年）到非闰年的情况。

#### 场景：闰年 2/29 加一年到非闰年
- **当** 当前到期日为 2024-02-29，周期为 1 年
- **那么** 下一个到期日必须为 2025-02-28（非闰年 2 月最多 28 天）

#### 场景：非闰年 2/28 加一年到闰年（anchorDay 为 29）
- **当** 当前到期日为 2025-02-28，周期为 1 年，anchorDay 为 29
- **那么** 下一个到期日必须为 2026-02-28（2026 非闰年，仍然钳位到 28）

### 需求：所有公历日期计算入口统一
系统中所有涉及公历月/年周期计算的代码路径，必须使用统一的日期计算函数，禁止直接使用 `Date.setMonth()` 或 `Date.setFullYear()` 进行周期推算。

#### 场景：前端到期日计算
- **当** 用户在创建/编辑订阅表单中选择起始日期和周期
- **那么** 系统必须使用统一函数计算到期日

#### 场景：后端自动续费
- **当** 定时任务检测到订阅已过期且开启自动续费
- **那么** 系统必须使用统一函数计算新的到期日

#### 场景：手动续订
- **当** 用户通过 API 手动续订订阅
- **那么** 系统必须使用统一函数计算新的到期日
